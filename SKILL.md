# Create Skill

## Overview

Interactive wizard for creating new OpenCode packages with proper project structure and structured metadata. It now supports both classic skill-only projects and first-stage knowledge-hybrid projects. It separates development (~/Project/Skills) from deployment (~/.config/opencode/skills) to enable version control, discoverability, and independent management.

## When to Use

- Creating a new OpenCode skill for the first time
- Want a standardized skill-only or knowledge-hybrid project structure
- Need structured metadata for category, tags, boundary, and maturity
- Need version control support for skills
- Following the "development-deployment separation" pattern

## Quick Start

### Installation

```bash
cd ~/Project/Skills/opencode-skill-create
./install.sh
```

This creates the `opencode-create-skill` command and installs the skill.

### Creating a New Skill

```bash
# Interactive mode (recommended for first time)
opencode-create-skill

# Quick mode (if you know the name)
opencode-create-skill git-workflow
```

## Project Structures Created

```
opencode-skill-{name}/
├── SKILL.toml            # Canonical machine-readable manifest
├── SKILL.md              # Main skill instructions (installed to ~/.config/opencode/skills/)
├── install.sh            # Installs skill package files to ~/.config/opencode/skills/
├── uninstall.sh          # Removes skill from ~/.config/opencode/skills/
├── README.md             # Project documentation (not installed)
└── .git/                 # Version control
```

When `knowledge-hybrid` mode is selected, the wizard also generates:

```text
commands/
scripts/
templates/
```

These directories establish a first-stage contract for namespaced `save/search/load` workflow entrypoints plus a starter knowledge template.

## Skill Naming Rules

Valid skill names must:
- Use only lowercase letters, numbers, and hyphens
- Start with a letter or number
- Not contain spaces or special characters
- Be under 50 characters

**Examples:**
- ✅ `git-workflow`
- ✅ `code-review`  
- ✅ `session-journal`
- ❌ `Git Workflow` (spaces)
- ❌ `code_review` (underscore)
- ❌ `my-skill-2024!` (special character)

## Skill Types

The wizard will ask you to choose a type:

| Type | Description | Use Case |
|------|-------------|----------|
| **Technique** | Step-by-step method | Concrete process with clear steps |
| **Pattern** | Mental model | Way of thinking about problems |
| **Reference** | API docs, guides | Documentation-heavy content |
| **Discipline** | Rules to follow | Enforcing standards/requirements |

## Skill Metadata

The generator now writes canonical metadata to `SKILL.toml` and keeps `SKILL.md` focused on human-readable instructions.

`tags` and `topics` remain required metadata fields, but they are generated by default from the skill inputs and then confirmed or edited by the author.

### Required manifest metadata

```toml
manifest_version = "1.0"

[skill]
name = "your-skill-name"
version = "0.1.0"
description = "Use when specific triggering conditions"
type = "technique|pattern|reference|discipline"
category = "engineering|frontend|backend|devops|testing|security|research|writing|workflow|product|general"
tags = ["search-keyword"]
topics = ["higher-level-theme"]
boundary = "One sentence describing what this skill covers"
maturity = "draft|alpha|beta|stable|deprecated"
last_verified = "2026-03-14"
```

### Optional manifest metadata

```toml
non_goals = ["Problem this skill intentionally does not solve"]
```

### Why this metadata exists

- **Category** gives every skill a primary home for management.
- **Tags/topics** are required so every skill has both keyword-level and theme-level discoverability.
- **Boundary** keeps the skill focused instead of letting it become vague.
- **Non-goals** make misuse less likely.
- **Maturity** provides a lightweight quality signal without pretending to assign an objective score.

## Environment Variables

```bash
# Optional: Override default development directory
export OPENCODE_SKILLS_DEV_DIR="$HOME/Project/Skills"
```

Default is `$HOME/Project/Skills` if not set.

## Installation Script Behavior

The generated `install.sh`:
1. Creates `~/.config/opencode/skills/{skill-name}/`
2. Copies `SKILL.toml` and `SKILL.md`
3. Copies optional runtime directories when present (`supporting/`, `commands/`, `scripts/`, `templates/`, `docs/`)

Mutable knowledge data should still live outside the installed runtime package.

## Hybrid Manifest and Data Contract

For `knowledge-hybrid` projects, the generator writes:

- `[hybrid]` metadata for namespace and generated commands
- `[knowledge]` metadata for storage format and path contract

The default data-dir contract is designed to be package-manager friendly:

```bash
${XDG_DATA_HOME:-$HOME/.local/share}/opencode/<namespace>
```

This keeps runtime package files deployable while allowing knowledge entries to remain mutable in a user data directory.

## Workflow

```
Create Skill → Edit Content → Install → Test → Git Commit
```

1. **Create**: Run `opencode-create-skill` to generate project
2. **Edit**: Modify SKILL.md with your content
3. **Install**: Run `./install.sh` to activate skill
4. **Test**: Verify skill works in OpenCode
5. **Commit**: `git add . && git commit -m "Add initial skill"`

## Common Mistakes

### Mistake 1: Editing files in ~/.config/opencode/skills/
**Problem**: Changes will be lost on reinstall
**Fix**: Always edit in `~/Project/Skills/`, then reinstall

### Mistake 2: Wrong skill name format
**Problem**: OpenCode won't recognize the skill
**Fix**: Use only lowercase letters, numbers, and hyphens

### Mistake 3: Not using "Use when" in description
**Problem**: OpenCode can't determine when to load the skill
**Fix**: Always start description with "Use when..."

### Mistake 4: Including too much in SKILL.md
**Problem**: Burns context window unnecessarily
**Fix**: Keep skills under 500 words; move heavy reference to supporting files

## SKILL.md Template Structure

The wizard generates a template based on skill type. All include metadata plus a focused content scaffold:

```yaml
---
name: your-skill-name
description: Use when specific triggering conditions
type: technique
category: engineering
tags:
  - search-keyword
topics:
  - higher-level-theme
boundary: "Focused on the specific capability this skill provides"
maturity: draft
---
```
